## 1) 供餐角色拆分與每週供應日設定
- 將「麵食」從既有主食邏輯中獨立為單一角色，並新增可配置的每週供應日（星期一至星期日）。
- 主菜（或主食）同步新增「每週供應日」設定，且需與麵食設定互不干擾、可各自獨立運作。
- 系統預設值調整為：
  - 麵食：僅週三供應。
  - 主菜（主食）：週一、週二、週四、週五供應。
- 新增「每日配菜數量（不含純蔬配菜）」可調整設定，並支援按日獨立配置。
- 每日配菜數量預設：
  - 週一、週二、週四、週五：2 道。
  - 週三：1 道。
- 驗收重點：角色拆分後，調整麵食供應日不應影響主菜供應日，反之亦然。

## 2) 菜色新增備菜時間與每日總備菜時長上限
- 在菜色資料模型新增欄位「備菜時間（分鐘）」：
  - 型別：整數。
  - 預設值：0。
  - 建議限制：不可為負值。
- 新增「每日備菜時間總合上限」規則，計算該日所有已排入菜色之備菜時間總和。
- 每日上限需可獨立設定（可依週幾不同而調整），預設值為 60 分鐘。
- 驗收重點：排餐結果不得超過當日上限；若無可行組合，需提供清楚提示與可調整方向。

## 3) 每日配菜＋湯品之「含蛋白質」數量上限
- 新增規則：可設定每日「配菜與湯品中，蛋白質欄位非空」的合計上限。
- 規則需支援按日獨立設定。
- 預設值：每日上限 2。
- 目的：避免同日出現過多肉類/高蛋白料理，降低整體油膩感。
- 驗收重點：演算法在排配菜與湯品時，需一併檢查該上限。

## 4) 修正特定菜名刪除失敗（Method Not Allowed）
- 問題描述：菜名為 `高麗菜/木瓜` 時，於圖形化介面執行刪除會回傳 **Method Not Allowed**，導致無法刪除。
- 需排查方向：
  - 前端路由/請求方法是否正確（DELETE / POST override 等）。
  - 菜名含 `/` 是否造成 URL path 被錯誤切段（編碼、escape、後端 route 參數型態）。
  - API 對特殊字元菜名的識別方式是否一致（建議改用 ID 刪除，名稱僅作顯示）。
- 驗收重點：可穩定刪除含特殊字元（尤其 `/`）的菜名，且不影響既有刪除流程。

## 5) 全菜色新增「允許供應週幾」以支援特供菜彈性
- 為所有菜色新增可選設定：「允許出現的週幾」。
- 預設值：所有週幾皆允許。
- 應用情境：
  - 例如「雞翅」雖分類為主菜，但肉量較少，可透過此設定限制僅在麵食特供日（如週三）才有機會被排入。
- 設計原則：
  - 以通用規則支援未來更多「特定菜僅限特定日」需求。
  - 與角色供應日（麵食/主菜）規則可疊加判斷，避免互相覆寫。
- 驗收重點：使用者可透過 UI 直接配置單一道菜的允許週幾，並實際反映在排餐結果中。
